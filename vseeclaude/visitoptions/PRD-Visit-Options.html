1. Overview & Purpose

The Visit Options & Waiting Room Settings module is the administrative control surface in VSee Clinic that lets clinic administrators configure the full patient-facing visit experience. It governs how patients discover a waiting room, what visit types are available, what intake information is collected, how payments are handled, and what clinical note templates providers use.

The V2 architecture introduces a two-tier hierarchy: clinic-level settings (payment configuration, reusable templates) that are shared across all waiting rooms, and room-level settings that reference or override clinic defaults. This eliminates duplication and ensures configuration changes propagate consistently.

Key Principle: Configure once at the clinic level, customize per room only when needed. Visit options inherit clinic defaults for intake and notes templates unless explicitly overridden.

2. Goals & Success Metrics

Product Goals

  • Enable clinic admins to fully configure waiting rooms and visit types without engineering support.
  • Support complex payment scenarios across three patient types: self-pay, group-covered, and insurance.
  • Provide a live patient-facing preview so admins can verify the booking flow before publishing.
  • Reduce configuration errors through validation, smart defaults, and visual warnings for incomplete setup.

Success Metrics

MetricTarget
Time to configure a new waiting room from scratch< 5 minutes
Admin support tickets related to room configurationReduce by 40%
Visit options with incomplete payment setup (warning state)< 10% at any time
Adoption of clinic-level templates (vs. per-room overrides)> 70% of visit options using defaults

5. Feature: Clinic-Level Settings

Centralized configuration that applies across all waiting rooms. Accessible via the "Clinic Settings" nav item.

5.1 Payment Configuration

Defines how payments are collected for each patient type. These settings govern the payment options available when configuring individual visit options.

Patient TypeConfiguration
Self-Pay Accept payments toggle, Stripe connection status, payment timing (before/after/choice), default price, no-show fee (toggle + amount).
Group-Covered Require group ID toggle, require referral toggle, verification status access rules (verified / not verified / pending / error → allow / block / review).
Insurance Eligibility status access rules (eligible / not eligible / pending / error → allow / block / review).

5.2 Form Library

Reusable form templates that define what patient information is collected. Referenced by workflow form steps.

  • Each form has a name and an ordered list of fields.
  • Fields support 8 types: Text Input, Text Area, Checkbox, Scale (0–3), Dropdown, Date Picker, File Upload, Section Header.
  • Fields can be marked as enabled/disabled and required/optional.
  • CRUD operations: create, edit, delete.

Form Editor — Drag-and-Drop Builder

The form editor uses a two-column layout:

  • Left: Field palette — draggable field type cards (with icons). Drag a type onto the form to add it at any position.
  • Right: Fields list — existing fields with drag handles for reordering. Each field row shows an icon, label input, type selector, required/enabled toggles, and a delete button.
  • Drop zones appear between fields during drag operations with "Drop here" indicators.
  • Empty state prompts users to drag from the palette.

5.3 Notes Templates

Reusable clinical note templates that pre-populate the provider's visit notes.

Notes Editor — Section-Based Builder

The notes template editor uses a drag-and-drop section builder:

  • Section types: Section Heading, Text Area, Checkbox, Divider — each draggable from a left palette.
  • Existing sections can be reordered via drag handles.
  • Inline editing: headings render as bold text inputs, text areas render as textareas, checkboxes as label inputs, dividers as horizontal rules.
  • Presets: SOAP Note, Progress Note, and Blank quick-load structured sections.
  • On save, sections are serialized to both a structured sections[] array and a plain-text content string for backward compatibility.
  • Same default/override inheritance model as form templates.
Shared Editors: Both FormLibraryEditor and NotesTemplateEditor are extracted into a shared TemplateEditors.jsx module. They are used in Clinic Settings for template management and inline within the Visit Option modal for per-visit editing.

6. Feature: Intake Flow Customizer

A visual flow-chart-style builder that defines the patient journey through the clinic's intake process. Configured at the clinic level as the default workflow, with per-visit-option overrides available.

6.1 Workflow Modules

Step types organized into 4 categories, matching the VSee API:

CategoryStep Types
Account Signup, Signup Verification, Login
Forms Form — a unified step type with a form picker that includes both built-in and custom forms:
  • Built-in: Intake Form, Guest Intake, Insurance Form, Guarantor, Pharmacy, Emergency Contact, Create Dependant, Cancel Intake Survey
  • Custom: Any form from the clinic's Form Library (see §5.2)
The form picker groups these into <optgroup> sections for clarity.
Steps Consultation (Visit Selection), Dependant List, Calendar Picker, Payment Options, Test Device, EMR, Setup Session, Confirmation, Walk-in Confirmation
Logic Conditional Branch (by patient type or insurance status)
Unified Form Model: All form types (intake, insurance, pharmacy, etc.) are treated as instances of the single "Form" step type. Built-in forms provide standard templates; custom forms come from the clinic's form library. This simplifies the step palette and avoids redundant step types.

6.2 Visual Flow Builder

  • Flow canvas: Vertical flow with "Patient Enters" start node, step cards connected by lines, and "Session Starts" end node.
  • Add Step: Full-width "Add step" button between each step. Opens a categorized picker dropdown.
  • Drag-and-drop reordering: Each step card has a drag handle. Drop zones appear between steps during drag with visual feedback.
  • Conditional branches: Branch on patient type or insurance status. Each branch is a column with its own step list. Steps can be dragged between branches.
  • Auto-pairing: Some steps auto-pair (e.g., Signup → Signup Verification).
  • Singleton enforcement: Certain steps (e.g., Consultation) can only appear once.

6.3 Workflow Architecture

  • The Visit Selection step marks the boundary: steps before it run for all patients; steps after it follow the selected visit option.
  • Individual visit options can override the post-selection workflow via workflowOverride.
  • workflowOverride: null = use clinic default workflow.
Conditional Branches: Branch columns expand to 820px max-width with 180px minimum per branch to prevent content clipping. Horizontal scroll is available as a fallback for many branches.

7. Feature: Waiting Rooms List

Displays all waiting rooms in a table view with summary information. Serves as the entry point for room management.

Capabilities

  • View: Table showing room name, code, visibility badge, patient type count, and enabled visit option count.
  • Add Room: Creates a new room with sensible defaults (unlisted, self-pay only, Pacific timezone, always open, random 5-char code).
  • Delete Room: Confirmation modal required. Cannot delete the last remaining room.
  • Navigate: Click any row or "Configure" button to open room settings.
  • Single-room shortcut: When only one room exists, the app skips the list view and goes directly to room settings.

8. Feature: Waiting Room Settings

The primary configuration page for an individual waiting room. Organized as a single-page form with distinct sections.

8.1 Room Identity

FieldTypeBehavior
Room NameText inputEditable. Displayed in breadcrumb and patient-facing UI.
Room CodeRead-only + copyAuto-generated 5-character alphanumeric code. Clipboard copy with toast confirmation.

8.2 Operating Hours

  • Timezone: Dropdown with 14 common timezone options.
  • Mode: "Always Open" or "Scheduled Hours."
  • Schedule: Per-day configuration (Sunday–Saturday). Each day can be marked open/closed with configurable time slots (30-min increments, 12:00 AM–11:30 PM). Multiple slots per day supported.
  • Closure Message: Custom text shown to patients when the room is closed.

8.3 Visibility

  • Public: Discoverable via search in the VSee Clinic mobile app.
  • Unlisted: Accessible only via direct link or room code.

8.4 Patient Types

Multi-select of which patient categories this room accepts:

  • Self-Pay — Patients paying out of pocket.
  • Group-Covered — Patients covered by an employer or organization group.
  • Insurance — Patients using health insurance.
Dependency: The patient types enabled at the room level constrain which types can be selected on individual visit options. Disabling a type at the room level removes it as an option for all visit options in that room.

8.5 Dirty State & Save

  • Changes are tracked via JSON serialization comparison.
  • "Unsaved changes" indicator appears when the form is dirty.
  • "Update Settings" button is disabled when no changes exist.
  • Success toast on save.

9. Feature: Visit Options

The core of the module. Visit options define the appointment types patients can book within a waiting room.

9.1 Visit Options Table

  • Columns: order (up/down buttons), name, duration + session type badge, mode (with icons), visible toggle, patient type badges, action buttons (edit, delete).
  • Up/down button reordering — simple arrow buttons to move visit options up or down in the list. Suitable for the typical case of < 5 visit options per room.
  • Inline visibility toggle — switch visit options on/off without opening the modal.
  • Payment warning indicator — amber triangle with "Set up payment" link when pricing is incomplete. Clicking opens the modal directly on the Payment tab.
  • Empty state with call-to-action when no visit options exist.

9.2 Visit Option Modal (Add / Edit)

A multi-tab modal for full visit option configuration. Four tabs: General, Patient Types, Intake Flow, Visit Notes.

General Tab

FieldTypeValidation
Visit NameText inputRequired. Error shown on empty.
DurationSelect15 / 30 / 45 / 60 / 90 / 120 min
Visit ModeToggle + multi-select pillsTwo-way toggle: Live Visit (Video, Phone, In-person — multi-select pills) or E-Consult (async). Cannot mix synchronous and asynchronous modes.
Session TypeSelect1:1 or Group
Slots (max)Number input1–99. Disabled and locked to 1 for 1:1 sessions.
Visible to PatientsToggleControls whether this option appears in the booking flow.
SpecialtiesCheckbox pillsMulti-select from clinic specialties. "Select All / Deselect All" toggle. Empty = visible to all.

Patient Types Tab

Accepted patient types and per-type payment configuration.

  • Accepted Patient Types: 3-column grid of square cards matching the clinic settings layout. Each card shows an icon, label, and description with a checkbox overlay. At least one required. Only types enabled at the room level are available.
  • Payment Collection: Per-patient-type pricing displayed as sub-tabs (one tab per selected patient type).
Patient TypePayment Methods Available
Self-Pay Radio selection: "No collection required" or "Collect specific amount" (with dollar input).
Group-Covered Status-based grid (Verified / Not Verified / Pending / Error). Per-status options: No collection, Collect copay (with fallback amount), Specific amount. Statuses blocked at clinic level are grayed out.
Insurance Status-based grid (Eligible / Not Eligible / Pending / Error). Same per-status options as Group-Covered. Statuses blocked at clinic level are grayed out.
Completeness Check: The system runs isPaymentIncomplete() to detect visit options where a payment method is set to "specific" but no amount is entered, or where copay is set but the status is not blocked. Incomplete items show a warning badge in the table.

Intake Flow Tab

  • Full Workflow Customizer instance embedded in the modal for per-visit-option workflow override.
  • By default, inherits the clinic's default workflow (workflowOverride: null).
  • When overridden, provides the same visual flow builder with drag-and-drop and conditional branching.
  • The Form step type uses a unified picker with built-in forms (Intake, Insurance, Pharmacy, etc.) and custom forms from the clinic library — all forms are selected from a single dropdown grouped by category (see §6.1).
  • Includes a Patient Preview panel that simulates the workflow in a mobile phone frame.

Visit Notes Tab

  • Same inheritance model as Intake Flow: clinic default toggle + override dropdown.
  • When a specific template is selected, an Edit button opens the NotesTemplateEditor inline within the modal — the same drag-and-drop section builder used in Clinic Settings (see §5.3).
  • Preview shows the template content in a read-only monospace textarea when not editing.

9.3 Delete Visit Option

Triggered from the table action column. Requires confirmation via modal. Irreversible.

10. Feature: Patient Preview

A side-panel simulation of the patient booking flow, toggled from the room settings header.

Behavior

  • Activating the preview switches the layout to a two-column grid (settings left, 400px preview panel right).
  • Preview renders a mobile-phone-style simulation of the patient-facing experience.
  • Reflects the current unsaved state in real time — changes in the settings form are immediately visible in the preview.

Preview Steps (Simulated Flow)

The preview walks through the workflow steps defined in the intake flow customizer. Each step type renders a fully interactive patient-facing simulation:

  • Login / Signup / Verification: Simulated auth screens with form fields.
  • Dependant List: Member selection with "Myself" option and dependant cards.
  • Visit Selection: Shows all visible visit options with name, duration, mode icons, and group slot count.
  • Calendar Picker: Interactive date/time slot selection grid.
  • Form (all types): Renders the selected form — built-in (Intake, Guest Intake, Insurance, Guarantor, Pharmacy, Emergency Contact) or custom from the form library. Respects field type, enabled/required flags.
  • Payment: Credit card selection with pricing based on patient type and status.
  • Test Device: Camera/microphone permission simulation.
  • Setup Session / EMR: Loading/processing step simulations.
  • Confirmation / Walk-in Confirmation: Summary screen with visit details.
  • Conditional Branch: Automatically routes to the correct branch based on patient type or insurance status.

12. Component Architecture

App.jsx └── VersionBLayout (router / state owner) ├── ClinicTemplatesPage (clinic settings) │ ├── PatientTypes │ ├── PaymentSettings │ ├── WorkflowCustomizer (intake flow builder) │ │ ├── StepCard / Connector / AddStepPicker │ │ └── ConditionalBranches │ ├── FormLibraryEditor ←─┐ │ └── NotesTemplateEditor ←┤ (from TemplateEditors.jsx) ├── WaitingRoomsListPage │ (room list / CRUD) └── WaitingRoomSettingsV2 │ (single room config) ├── Breadcrumb │ ├── OperatingHours │ ├── PatientTypes │ ├── VisitOptionsTableV2 │ (up/down reorder) │ ├── RoomVisitOptionModal │ │ ├── NotesTemplateEditor ←─┘ (shared editors) │ │ └── WorkflowCustomizer │ └── ConfirmModal (delete confirmation) └── PatientPreview (live patient simulation) └── 15+ step components (one per workflow step type)

State Management

  • VersionBLayout owns the top-level state: clinic object and rooms[] array.
  • Room settings page uses a useDirty() custom hook that tracks a serialized "saved" snapshot for dirty detection.
  • All state is held in React's useState — no external state management library.
  • Data flows downward via props; changes flow upward via callback props (onChange, onSave).

15. Future Considerations

  • API integration & persistence: Current implementation is client-side state only. Needs backend API for CRUD operations with optimistic UI updates.
  • Role-based access control: Differentiate between clinic admin and provider-level permissions for room and template management.
  • Custom intake fields: Done Form library now supports 8 field types with drag-and-drop builder. Future: add conditional visibility rules between fields.
  • Multi-provider rooms: Support assigning multiple providers to a single waiting room with provider-specific visit options.
  • Scheduling/calendar integration: Connect visit options to provider availability calendars for real-time slot booking.
  • Stripe payment gateway: Complete the Stripe Connect integration (currently mocked) for actual payment collection.
  • Audit log: Track configuration changes with timestamps and user attribution.
  • Bulk operations: Copy visit options between rooms, duplicate room configurations.
  • Localization: Support multi-language configuration for patient-facing content (room names, visit option names, closure messages).
