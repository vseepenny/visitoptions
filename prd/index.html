1. Overview & Purpose

The Visit Options & Waiting Room Settings module is the administrative control surface in VSee Clinic that lets clinic administrators configure the full patient-facing visit experience. It governs how patients discover a waiting room, what visit types are available, what intake information is collected, how payments are handled, and what clinical note templates providers use.

The V2 architecture introduces a two-tier hierarchy: clinic-level settings (payment configuration, reusable templates) that are shared across all waiting rooms, and room-level settings that reference or override clinic defaults. This eliminates duplication and ensures configuration changes propagate consistently.

Key Principle: Configure once at the clinic level, customize per room only when needed. Visit options inherit clinic defaults for intake and notes templates unless explicitly overridden.

2. Goals & Success Metrics

Product Goals

  • Enable clinic admins to fully configure waiting rooms and visit types without engineering support.
  • Support complex payment scenarios across three patient types: self-pay, group-covered, and insurance.
  • Provide a live patient-facing preview so admins can verify the booking flow before publishing.
  • Reduce configuration errors through validation, smart defaults, and visual warnings for incomplete setup.

Success Metrics

MetricTarget
Time to configure a new waiting room from scratch< 5 minutes
Admin support tickets related to room configurationReduce by 40%
Visit options with incomplete payment setup (warning state)< 10% at any time
Adoption of clinic-level templates (vs. per-room overrides)> 70% of visit options using defaults

5. Core User Flows

The following user flows describe end-to-end journeys through the module. Each flow is written from the perspective of a clinic admin performing a common task.

5.1 First-Time Clinic Setup

A new clinic admin configures their clinic from scratch for the first time.

Clinic Settings Waiting Room Settings ───────────── ───────────────────── 1. Navigate to "Clinic Settings" 2. Patient Types tab → Enable patient types (e.g. Self-Pay, Insurance) → Configure payment rules per type 3. Intake Flow tab → Select a workflow template (e.g. "Standard Telehealth") → Customize steps as needed → Save 4. Notes Templates tab → Create notes template (e.g. SOAP Note) → Set as default → Save 5. Navigate to "Waiting Rooms" 6. Click into the default room → Set room name & operating hours → Set visibility (Public / Unlisted) 7. Add first visit option → Fill General tab (name, duration, mode) → Select patient types → Verify inherited intake flow → Verify inherited notes template → Save 8. Click "Patient Preview" to verify 9. Save room settings
Key Decision Points: The admin decides at step 3 whether the clinic default workflow is sufficient, or whether individual visit options will need overrides (step 7). Most clinics use the default for 70%+ of visit options.

5.2 Add a New Visit Option

An admin adds a new visit type to an existing room.

  1. Navigate to Waiting Rooms → select room.
  2. In the Visit Options section, click "Add Visit Option".
  3. General tab: Enter visit name, select duration, choose visit mode (Video / Phone / In-person / E-Consult), set session type (1:1 or Group).
  4. Patient Types tab: Select accepted patient types. Configure per-type payment amounts.
  5. Intake Flow tab: By default, "Use clinic default" is on. Toggle off to customize:
    • Select a workflow template or start from blank.
    • Add/remove/reorder steps using the visual flow builder.
    • Add conditional branches if needed (e.g. branch by patient type).
    • Optionally save the custom workflow as a reusable template.
  6. Visit Notes tab: Use clinic default or select a specific notes template. Edit inline if needed.
  7. Click "Save Visit Option".
  8. Back on the room page, reorder the new option using the up/down pill buttons.
  9. Use the Patient Preview to verify the booking flow.

5.3 Configure Payment for Multiple Patient Types

An admin sets up payment collection for a visit option that accepts Self-Pay, Insurance, and Group-Covered patients.

  1. Open the visit option modal → Patient Types tab.
  2. Select all three patient types.
  3. Payment sub-tabs appear — one per patient type:
    • Self-Pay tab: Choose "Collect specific amount" → enter $150.
    • Insurance tab: For each verification status (Eligible / Not Eligible / Pending / Error), choose collection method:
      • Eligible → Collect copay ($25, fallback $50)
      • Not Eligible → Collect specific amount ($150)
      • Pending → No collection
      • Error → No collection
    • Group-Covered tab: Configure per-status rules similarly.
  4. Save. If any "specific amount" field is left blank, an amber warning appears in the visit options table.
Incomplete Payment Warning: The table shows an amber triangle with "Set up payment" link next to any visit option with incomplete pricing. Clicking it opens the modal directly on the Payment tab.

5.4 Build a Custom Intake Flow with Branching

An admin creates a workflow that branches by patient type — sending Self-Pay and Insurance patients through different paths.

  1. Navigate to Clinic Settings → Intake Flow tab (or open a visit option's Intake Flow tab and toggle off "Use clinic default").
  2. Click the template name button → select "Standard Telehealth" template from the modal.
  3. Template applies with pre-configured steps. Customize:
    • Click the "+" button between steps to add a new step.
    • Select Logic → Conditional Branch.
    • Choose branch condition: "Patient Type".
  4. Configure each branch using the tabbed interface:
    • Click the Self-Pay tab → add "Payment Options" step.
    • Click the Insurance tab → add "Insurance Form" → "Guarantor" steps.
    • Click the Group-Covered tab → add relevant steps.
  5. Reorder steps within branches using the joined up/down pill buttons.
  6. Expand any step card (click + icon) to see and edit configuration details.
  7. Click "Save as Template" to save this workflow for reuse in other visit options.
  8. Save settings.

5.5 Use and Manage Workflow Templates

An admin applies a template and saves a customized version for reuse.

  1. In the intake flow editor, click the template name button to open the template modal (list on the left, preview on the right).
  2. Browse built-in templates (Standard Telehealth, Quick Walk-in, Scheduled Visit, Insurance-First, Self-Pay Only, Blank) and any saved custom templates. Use the search bar to filter by name or description.
  3. Click a template in the list to preview its full step flow — including branch contents — without applying it.
  4. Click "Use This Template" in the preview footer to apply it. The footer warns that this replaces the current workflow (all step IDs are regenerated).
  5. Customize the applied template as needed.
  6. Click "Save as Template":
    • The modal opens in save mode with a name input.
    • If the workflow came from a custom template, choose Update "<name>" (overwrite in place) or Save as a new template.
    • Enter a name (e.g. "Dermatology Intake") and save.
    • New templates appear in the Custom section of the template modal.
  7. Custom templates can be renamed (pencil icon) or deleted (trash icon) from the preview pane header.
Template Availability: Custom templates saved from Clinic Settings or from a visit option's intake flow editor are stored at the clinic level and available across all rooms and visit options.

5.6 Override Clinic Defaults for a Specific Visit Option

An admin customizes the intake flow and notes template for a single visit option, leaving all others on clinic defaults.

  1. Open the visit option modal → Intake Flow tab.
  2. The clinic default workflow is shown as a read-only preview.
  3. Toggle off "Use clinic default intake flow".
  4. The clinic workflow is cloned into an editable copy. Modify as needed.
  5. Switch to Visit Notes tab.
  6. Toggle off "Use clinic default" → select a specific notes template from the dropdown.
  7. Click "Edit" to modify the template inline using the form-editor-style builder — click or drag from the 8-type section palette (headings, text, inputs, checkboxes, scales, dropdowns, dates, dividers). Saves apply clinic-wide.
  8. Save the visit option. Only this visit option uses the overrides; all others continue using clinic defaults.

5.7 Manage Multiple Waiting Rooms

An admin sets up and manages multiple rooms for different use cases (e.g. General Practice room and Mental Health room).

  1. Navigate to Waiting Rooms list page.
  2. Click "Add Room" — a new room is created with defaults (unlisted, self-pay, always open, random code).
  3. Click into the new room to configure:
    • Set room name (e.g. "Mental Health Clinic").
    • Configure operating hours (scheduled, Mon–Fri 9–5).
    • Set visibility to Public.
    • Add visit options specific to this room.
  4. Use Patient Preview to verify the patient experience for this room.
  5. Navigate back to the rooms list via breadcrumb.
  6. The list shows a summary: room name, code, visibility badge, patient type count, and visit option count.
  7. To remove a room, click the delete icon (confirmation required). The last remaining room cannot be deleted.

5.8 Verify Configuration with Patient Preview

An admin uses the live preview to verify the patient-facing booking experience before publishing.

  1. On the room settings page, click "Patient Preview" button in the header.
  2. The layout splits: settings on the left, mobile-frame preview on the right.
  3. The preview simulates the full patient journey step by step:
    • Login / Signup screens.
    • Visit option selection (shows all visible options with name, duration, mode icons).
    • Calendar/scheduling step (if present).
    • Intake forms — renders actual fields from the form library.
    • Payment step — shows pricing based on patient type and verification status.
    • Device test, session setup, and confirmation screens.
  4. Changes made in the settings panel reflect immediately in the preview (real-time, no save required).
  5. Toggle patient type or insurance status in the preview to test different branching paths.
  6. Click "Hide Preview" to return to the full-width settings view.

6. Feature: Clinic-Level Settings

Centralized configuration that applies across all waiting rooms. Accessible via the "Clinic Settings" nav item.

6.1 Payment Configuration

Defines how payments are collected for each patient type. These settings govern the payment options available when configuring individual visit options.

Patient TypeConfiguration
Self-Pay Accept payments toggle, Stripe connection status, payment timing (before/after/choice), default price, no-show fee (toggle + amount).
Group-Covered Require group ID toggle, require referral toggle, verification status access rules (verified / not verified / pending / error → allow / block).
Insurance Eligibility status access rules (eligible / not eligible / pending / error → allow / block).

6.2 Form Library

Reusable form templates that define what patient information is collected. Referenced by workflow form steps.

  • Each form has a name and an ordered list of fields.
  • Fields support 8 types: Text Input, Text Area, Checkbox, Scale (0–3), Dropdown, Date Picker, File Upload, Section Header.
  • Fields can be marked as enabled/disabled and required/optional.
  • CRUD operations: create, edit, delete.

Form Editor — Drag-and-Drop Builder

The form editor uses a two-column layout:

  • Left: Field palette — draggable field type cards (with icons). Drag a type onto the form to add it at any position.
  • Right: Fields list — existing fields with drag handles for reordering. Each field row shows an icon, label input, type selector, required/enabled toggles, and a delete button.
  • Drop zones appear between fields during drag operations with "Drop here" indicators.
  • Empty state prompts users to drag from the palette.

6.3 Notes Templates

Reusable clinical note templates that pre-populate the provider's visit notes.

Notes Editor — Same Builder UI as the Form Editor

The notes template editor uses the same field-based builder UI as the Form Library editor — palette on the left, typed rows on the right:

  • Section types (8): Section Heading, Text Area, Text Input, Checkbox, Scale (0–3), Dropdown, Date Picker, Divider — click or drag from the left palette to add (click appends, drag positions).
  • Row anatomy matches the form editor: drag handle, type icon, label/content input, type dropdown (rows can be converted between types in place), and remove button.
  • Existing sections can be reordered via drag handles.
  • Presets: SOAP Note, Progress Note, and Blank quick-load structured sections.
  • On save, sections are serialized to both a structured sections[] array and a plain-text content string for backward compatibility (structured types flatten to text placeholders, e.g. scale → Label: 0 · 1 · 2 · 3).
  • Same default/override inheritance model as form templates.

Structured Notes Preview

  • Wherever a notes template is previewed (Visit Notes tab of the visit option modal, clinic-default preview), it renders as a structured read-only mock of the provider note — bold headings, empty entry boxes, checkbox rows, 0–3 scale pills, dropdown and date field mocks, dividers — instead of the previous raw monospace textarea.
  • Legacy content-string templates are line-parsed into headings/text for rendering; structured sections[] templates render each typed section natively.
Shared Editors: Both FormLibraryEditor and NotesTemplateEditor are extracted into a shared TemplateEditors.jsx module. They are used in Clinic Settings for template management and inline within the Visit Option modal for per-visit editing.

6.4 Landing Page (Clinic Front Door)

Configures what patients see when they open the clinic link, before choosing a waiting room. Edited in the Landing Page tab of Clinic Settings with a live patient-view preview (browser-chrome mock) beside the form.

  • Branding: clinic display name, tagline, brand color (6 preset swatches) — applied to the header of the clinic front door and every room landing page.
  • Welcome message: the default greeting text; rooms inherit it unless they override.
  • Support contact: email + phone shown in the landing page footer.
  • Toggles: show room directory (lists public rooms only — unlisted rooms never appear), show operating hours.
  • All fields update the preview live; stored in clinic.landingPage and saved with the standard Clinic Settings dirty/save flow.
  • Preview devices: a Web / Mobile app segmented toggle switches the preview between browser chrome and a phone shell (status bar, brand app bar, home indicator). An expand button opens the preview in a larger modal (~720px web / larger phone), with the same device toggle inside; Esc, backdrop, or ✕ closes. Applies to both the clinic and room preview panels.

7. Feature: Intake Flow Customizer

A visual flow-chart-style builder that defines the patient journey through the clinic's intake process. Configured at the clinic level as the default workflow, with per-visit-option overrides available.

7.1 Workflow Modules

Step types organized into 4 categories, matching the VSee API:

CategoryStep Types
Account Signup, Signup Verification, Login
Forms Form — a unified step type with a form picker that includes both built-in and custom forms:
  • Built-in: Intake Form, Guest Intake, Insurance Form, Guarantor, Pharmacy, Emergency Contact, Create Dependant, Cancel Intake Survey
  • Custom: Any form from the clinic's Form Library (see §6.2)
The form picker groups these into <optgroup> sections for clarity.
Steps Consultation (Visit Selection), Dependant List, Calendar Picker, Payment Options, Test Device, EMR, Setup Session, Confirmation, Walk-in Confirmation
Logic Conditional Branch (by patient type or insurance status)
Unified Form Model: All form types (intake, insurance, pharmacy, etc.) are treated as instances of the single "Form" step type. Built-in forms provide standard templates; custom forms come from the clinic's form library. This simplifies the step palette and avoids redundant step types.

7.2 Visual Flow Builder

  • Flow canvas: Vertical flow with "Patient Enters" start node, step cards connected by lines, and "Session Starts" end node.
  • Add Step: Full-width "Add step" button between each step. Opens a categorized picker dropdown.
  • Drag-and-drop reordering: Each step card has a drag handle. Drop zones appear between steps during drag with visual feedback. Step card actions: joined up/down pill on the left, expand/collapse (±) and delete (×) on the right.
  • Conditional branches: Branch on patient type or insurance status. Uses a tabbed layout — each branch condition is a tab with colored indicator dot and step count badge. Active tab shows its step list at full width. Steps can be dragged between branches by switching tabs.
  • Auto-pairing: Some steps auto-pair (e.g., Signup → Signup Verification).
  • Singleton enforcement: Certain steps (e.g., Consultation) can only appear once.

7.3 Workflow Architecture

  • The Visit Selection step marks the boundary: steps before it run for all patients; steps after it follow the selected visit option.
  • Individual visit options can override the post-selection workflow via workflowOverride.
  • workflowOverride: null = use clinic default workflow.
Conditional Branches: Rendered as a tabbed interface rather than side-by-side columns. Each branch condition is a tab; the active tab's step list gets the full available width. This scales well to 3–4+ branches without cramping. Tab bar scrolls horizontally if needed.

7.4 Workflow Template Library

Pre-built and custom workflow templates provide starting points for intake flow configuration. Users select a template, then customize from there.

Template Selection Modal — Master-Detail with Preview

  • Trigger: Clicking the template name button opens a wide modal (~880px) with a master-detail layout: template list on the left, preview pane on the right.
  • Template list (left): Search bar on top; compact rows (icon, name, step count) grouped into Built-in and Custom sections. A checkmark marks the currently applied template. "Save Current as Template" button pinned at the bottom.
  • Preview pane (right): Selecting a list item previews it — nothing is applied yet. Shows name, Custom/Applied badges, description, step count, branching indicator, and a read-only mini flow (Patient Enters → step cards with icons → Session Starts). Conditional branches render as labeled groups with nested sub-steps; empty branches read "No steps — continues directly".
  • Applying: Explicit "Use This Template" button in the preview footer applies the template (stamps fresh IDs on all steps) and closes the modal. The footer warns "Applying replaces your current workflow (N steps)"; if the previewed template is already applied, the button reads "Re-apply Template" with a reset warning instead.
  • Search: Filters by template name and description.

Save, Rename, Update & Delete Custom Templates

  • "Save as Template" button appears next to the template selector when the workflow has steps and the onSaveTemplate callback is provided.
  • Opens the modal in save mode with a name input (pre-filled with the workflow name), step count summary, and Save/Cancel actions.
  • Update in place: If the current workflow was applied from a custom template, save mode offers a "Save to" choice — Update "<name>" (overwrites the existing template's steps, default) or Save as a new template. The confirmation copy and button label ("Update Template" / "Save Template") follow the selection.
  • Rename: A pencil icon in the preview pane header (custom templates only) swaps the title for an inline input — Enter saves, Escape cancels. Renaming the currently applied template also updates the workflow's displayed name.
  • Delete: A trash icon in the preview pane header removes the custom template; the preview falls back to the first built-in template.
  • Saved templates appear in the Custom section and are stored in clinic.workflowTemplates[].
  • All of the above is available in both Clinic Settings and the visit option modal's Intake Flow tab — templates saved from either location are stored at the clinic level and shared across all rooms.

8. Feature: Waiting Rooms List

Displays all waiting rooms in a table view with summary information. Serves as the entry point for room management.

Capabilities

  • View: Table showing room name, code, visibility badge, patient type count, and enabled visit option count.
  • Add Room: Creates a new room with sensible defaults (unlisted, self-pay only, Pacific timezone, always open, random 5-char code).
  • Delete Room: Confirmation modal required. Cannot delete the last remaining room.
  • Navigate: Click any row or "Configure" button to open room settings.
  • Single-room shortcut: When only one room exists, the app skips the list view and goes directly to room settings.

9. Feature: Waiting Room Settings

The primary configuration page for an individual waiting room. Organized as a single-page form with distinct sections.

9.1 Room Identity

FieldTypeBehavior
Room NameText inputEditable. Displayed in breadcrumb and patient-facing UI.
Room CodeRead-only + copyAuto-generated 5-character alphanumeric code. Clipboard copy with toast confirmation.

9.2 Operating Hours

  • Timezone: Dropdown with 14 common timezone options.
  • Mode: "Always Open" or "Scheduled Hours."
  • Schedule: Per-day configuration (Sunday–Saturday). Each day can be marked open/closed with configurable time slots (30-min increments, 12:00 AM–11:30 PM). Multiple slots per day supported.
  • Closure Message: Custom text shown to patients when the room is closed.

9.3 Visibility

  • Public: Discoverable via search in the VSee Clinic mobile app.
  • Unlisted: Accessible only via direct link or room code.

9.4 Patient Types

Multi-select of which patient categories this room accepts:

  • Self-Pay — Patients paying out of pocket.
  • Group-Covered — Patients covered by an employer or organization group.
  • Insurance — Patients using health insurance.
Dependency: The patient types enabled at the room level constrain which types can be selected on individual visit options. Disabling a type at the room level removes it as an option for all visit options in that room.

9.5 Landing Page (Room)

A section on the room settings page controlling what patients see when they open the room's direct link, with a live patient-view preview.

  • Inheritance: "Use clinic default welcome" toggle (on by default, landingPage: null). When on, the room shows its room name as the heading with the clinic welcome message. Branding (name, color, support contact) always comes from Clinic Settings — rooms cannot fork the brand.
  • Override: toggling off pre-fills a room-specific welcome heading and message (e.g. "Mental Health & Wellness" with a confidentiality reassurance), plus toggles to show/hide the visit options list and operating hours.
  • The preview renders the real room data: visible visit options as bookable rows, the room URL (vsee.me/<code>), and an hours line reflecting the room's Always Open / Scheduled mode.
  • Saved with the room's standard dirty/save flow.

9.6 Dirty State & Save

  • Changes are tracked via JSON serialization comparison.
  • "Unsaved changes" indicator appears when the form is dirty.
  • "Update Settings" button is disabled when no changes exist.
  • Success toast on save.

10. Feature: Visit Options

The core of the module. Visit options define the appointment types patients can book within a waiting room.

10.1 Visit Options Table

  • Columns: order (up/down buttons), name, duration + session type badge, mode (with icons), visible toggle, patient type badges, action buttons (edit, delete).
  • Pill-style reorder buttons — joined up/down button group (side-by-side pill) per row. Disabled state (grey) at top/bottom boundaries. Suitable for the typical case of < 5 visit options per room.
  • Inline visibility toggle — switch visit options on/off without opening the modal.
  • Payment warning indicator — amber triangle with "Set up payment" link when pricing is incomplete. Clicking opens the modal directly on the Payment tab.
  • Empty state with call-to-action when no visit options exist.

10.2 Visit Option Modal (Add / Edit)

A multi-tab modal for full visit option configuration. Four tabs: General, Patient Types, Intake Flow, Visit Notes.

General Tab

FieldTypeValidation

Patient Types Tab

Accepted patient types and per-type payment configuration.

  • Accepted Patient Types: 3-column grid of square cards matching the clinic settings layout. Each card shows an icon, label, and description with a checkbox overlay. At least one required. Only types enabled at the room level are available.
  • Payment Collection: Per-patient-type pricing displayed as sub-tabs (one tab per selected patient type).
Patient TypePayment Methods Available
Self-Pay Radio selection: "No collection required" or "Collect specific amount" (with dollar input).
Group-Covered Status-based grid (Verified / Not Verified / Pending / Error). Per-status options: No collection, Collect copay (with fallback amount), Specific amount. Statuses blocked at clinic level are grayed out.
Insurance Status-based grid (Eligible / Not Eligible / Pending / Error). Same per-status options as Group-Covered. Statuses blocked at clinic level are grayed out.
Completeness Check: The system runs isPaymentIncomplete() to detect visit options where a payment method is set to "specific" but no amount is entered, or where copay is set but the status is not blocked. Incomplete items show a warning badge in the table.

Intake Flow Tab

  • Full Workflow Customizer instance embedded in the modal for per-visit-option workflow override.
  • By default, inherits the clinic's default workflow (workflowOverride: null).
  • When overridden, provides the same visual flow builder with drag-and-drop and conditional branching.
  • The Form step type uses a unified picker with built-in forms (Intake, Insurance, Pharmacy, etc.) and custom forms from the clinic library — all forms are selected from a single dropdown grouped by category (see §7.1).
  • Includes a Patient Preview panel that simulates the workflow in a mobile phone frame.

Visit Notes Tab — Select from Library, Edit in One Place

  • Same inheritance model as Intake Flow: clinic default toggle + override dropdown.
  • Turning off "Use clinic default" reveals a dropdown of the clinic's notes template library — the visit option references a template; there is no per-visit-option template editing.
  • The selected template (or the dimmed clinic default) renders as a structured read-only preview of the provider note (see §6.3 Structured Notes Preview).
  • A hint under the preview points to Clinic Settings → Notes Templates, the single place templates are created and edited — an edit there applies everywhere the template is referenced.
  • Rationale: selection-only in the modal prevents the "I edited it for this visit but changed it clinic-wide" trap, and keeps visit options as lightweight references to the shared library (same principle as workflow templates).

10.3 Delete Visit Option

Triggered from the table action column. Requires confirmation via modal. Irreversible.

11. Feature: Patient Preview

A side-panel simulation of the patient booking flow, toggled from the room settings header.

Behavior

  • Activating the preview switches the layout to a two-column grid (settings left, 400px preview panel right).
  • Preview renders a mobile-phone-style simulation of the patient-facing experience.
  • Reflects the current unsaved state in real time — changes in the settings form are immediately visible in the preview.

Preview Steps (Simulated Flow)

The preview walks through the workflow steps defined in the intake flow customizer. Each step type renders a fully interactive patient-facing simulation:

  • Login / Signup / Verification: Simulated auth screens with form fields.
  • Dependant List: Member selection with "Myself" option and dependant cards.
  • Visit Selection: Shows all visible visit options with name, duration, mode icons, and group slot count.
  • Calendar Picker: Interactive date/time slot selection grid.
  • Form (all types): Renders the selected form — built-in (Intake, Guest Intake, Insurance, Guarantor, Pharmacy, Emergency Contact) or custom from the form library. Respects field type, enabled/required flags.
  • Payment: Credit card selection with pricing based on patient type and status.
  • Test Device: Camera/microphone permission simulation.
  • Setup Session / EMR: Loading/processing step simulations.
  • Confirmation / Walk-in Confirmation: Summary screen with visit details.
  • Conditional Branch: Automatically routes to the correct branch based on patient type or insurance status.

State Management

  • VersionBLayout owns the top-level state: clinic object and rooms[] array.
  • Room settings page uses a useDirty() custom hook that tracks a serialized "saved" snapshot for dirty detection.
  • All state is held in React's useState — no external state management library.
  • Data flows downward via props; changes flow upward via callback props (onChange, onSave).

16. Future Considerations

  • API integration & persistence: Current implementation is client-side state only. Needs backend API for CRUD operations with optimistic UI updates.
  • Review access option: Add a third access level ("Review") for Group-Covered verification and Insurance eligibility statuses — patient can book but requires staff approval before the visit is confirmed. Currently only Allow / Block are supported.
  • Role-based access control: Differentiate between clinic admin and provider-level permissions for room and template management.
  • Custom intake fields: Done Form library now supports 8 field types with drag-and-drop builder. Future: add conditional visibility rules between fields.
  • Multi-provider rooms: Support assigning multiple providers to a single waiting room with provider-specific visit options.
  • Scheduling/calendar integration: Connect visit options to provider availability calendars for real-time slot booking.
  • Stripe payment gateway: Complete the Stripe Connect integration (currently mocked) for actual payment collection.
  • Audit log: Track configuration changes with timestamps and user attribution.
  • Bulk operations: Copy visit options between rooms, duplicate room configurations.
  • Localization: Support multi-language configuration for patient-facing content (room names, visit option names, closure messages).
